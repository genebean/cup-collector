Adding Cups to the Catalog

The cup catalog is maintained in a spreadsheet and imported via a script. Here's the full workflow — from updating the spreadsheet to having new cups appear in the app.

The two roles in this workflow

  • Curator — maintains the spreadsheet, adds new cups, exports CSV
  • Importer — receives the CSV, runs the import script

The same person can do both. The split exists because the spreadsheet lives in Google Sheets or Excel, while the import script runs in a terminal.

Step 1 (Curator) — Update the spreadsheet

The spreadsheet has one row per cup, one column per field. The column names must match the CSV header exactly (case-insensitive). Required columns are marked below; all others are optional but recommended:

ColumnExampleNotes
nameSan FranciscoRequired. Display name — city, state, country, or themed location. Old CSVs using city as the column header are also accepted.
scopecityRecommended. "city" | "state" | "country" | "themed". Defaults to "city" if the column is absent or blank.
item_typemugOptional. "mug" | "ornament". Defaults to "mug" if absent.
regionCaliforniaOptional. State or province — leave blank if not applicable.
countryUnited StatesRequired. Full country name.
country_codeUSRequired. ISO 3166-1 alpha-2 code.
seriesYou Are HereRequired. Must be consistent — check existing entries for exact spelling.
year2021Required. 4-digit year.
lat37.7749Recommended. City/region centroid latitude — find via Google Maps. Without this, the cup won't appear on the map.
lng-122.4194Recommended. City/region centroid longitude.
image_urlhttps://example.com/cup.jpgOptional. Public image URL — downloaded and stored in PocketBase during import.
venue_seriesYou Are HereOptional. Themed cups only: the venue series name this cup is sold alongside.
hobbydb_urlhttps://hobbydb.com/...Optional. Link to the HobbyDB listing for reference.
more_info_urlhttps://...Optional. Any other reference URL.
notesLimited edition 2021Optional. Freeform notes shown on the cup detail screen.
Upsert key: name + series + year + item_type The import script matches records on the combination of name, series, year, and item_type. If a matching record exists it's updated; if not, it's created. This means the script is safe to re-run at any time without duplicating records — and that a mug and an ornament with the same name, series, and year are treated as separate records.

Step 2 (Curator) — Export as CSV

  1. In Google Sheets: File → Download → Comma Separated Values (.csv)
  2. In Excel: File → Save As → CSV (Comma delimited)
  3. Send or share the file with the importer.

Step 3 (Importer) — Run the import

There are two ways to run an import: the CLI script (recommended for large imports) or the web UI.

Option A — CLI (recommended)

Enter the dev shell first All commands must run inside nix develop. Never run npx or ts-node directly on the host. 
# 1. Enter the dev shell
nix develop

# 2. Preview what will change (no data is written)
import-cups --file /path/to/cups.csv --dry-run

# 3. Review the output — check for unexpected creates or updates

# 4. Apply the import
import-cups --file /path/to/cups.csv

The script prints a summary of created, updated, unchanged, and errored records. Run with --dry-run first whenever you're unsure about the data.

Default path for scraper-generated catalogs If you're importing the CSV produced by build-catalog (rather than a hand-maintained spreadsheet), the file lands at cups.csv in the project root by default and import-cups picks it up automatically — no --file flag needed. See Catalog Setup for the full scraper workflow.

Option B — Web UI

Open the app, go to Settings → Import Cups. Upload the CSV file, review the dry-run preview, then submit. The web UI uses the same upsert logic as the CLI script. Suitable for small updates; the CLI is faster and more informative for large imports.

Step 4 — Verify

  1. Open the PocketBase admin UI → Collections → cups — new entries should appear.
  2. Open the app — new cups should appear in Browse and on the map (if they have lat/lng coordinates).

Overriding scraped catalog data

If a cup's data on starbucks-mugs.com is wrong or incomplete (a missing image, an incorrect year, a note you want to add), you can fix it permanently using cups-overrides.csv in the project root. This is tracked in git and applied automatically at the end of every build-catalog run, so your corrections survive future scrapes without any manual re-patching.

The file uses the same columns as cups.csv. Only name and series are required to identify the cup — fill in only the fields you want to change:

name,series,image_url
Christmas Barcelona,You Are Here,https://example.com/xmas-barcelona.jpg

Blank cells in the overrides file are ignored and never clear a scraped value. After editing the file, re-run build-catalog then import-cups to apply the change.

The import-cups script also reports cups with no catalog image at the end of every run, so you always know which ones might benefit from an override entry.
