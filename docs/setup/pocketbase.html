PocketBase Setup

After deploying, PocketBase runs at https://pb.yourdomain.com. Access the admin UI at https://pb.yourdomain.com/_/ to verify the schema and create the household record.

Migrations run automatically
The migrations in pocketbase/migrations/ create all collections and set access rules on first start. If they applied cleanly, you only need Step 3 (create the household record). Check the PocketBase logs to confirm.

Step 1 — Access the admin UI

  1. Open https://pb.yourdomain.com/_/ in your browser.
  2. On first run, PocketBase prompts you to create an admin account. Use a strong password and store it in your password manager.
  3. Copy your email and password — you'll also need them in the .env file as POCKETBASE_ADMIN_EMAIL and POCKETBASE_ADMIN_PASSWORD so the app can write records server-side.

Step 2 — Verify collections

Go to Collections in the left sidebar. You should see three collections: households, cups, and owned_cups. If they're missing (migration didn't run), create them manually with these fields:

Step 2b — Verify access rules

The migration sets all three collections to allow unrestricted reads at the PocketBase level (empty rule string). This is intentional but does not mean the data is publicly accessible.

Security is enforced at the network layer: PocketBase is not exposed publicly — there is no nginx vhost for it by default in the NixOS module. All browser traffic reaches PocketBase through the Next.js /api/pb proxy, which requires a valid signed-in session before forwarding any request. The "unrestricted read" rule in PocketBase itself is only reachable by traffic that has already passed the session check in the proxy.

Browser clients do not hold PocketBase auth tokens (PocketBase's @request.auth rules can't be used because Auth.js handles identity, not PocketBase's own auth system). Enforcing access at the proxy layer achieves the same result cleanly.

Writes are admin-only in PocketBase. All mutations (marking cups owned, running imports) go through Next.js server-side API routes that authenticate with the admin email/password — the browser never gets write access to PocketBase directly.

Accessing the PocketBase admin UI
Without a public nginx vhost for PocketBase, use an SSH tunnel to reach the admin UI:

ssh -L 8090:localhost:8090 yourserver
then open http://localhost:8090/_/ in your browser.

Step 3 — Create the household record

  1. In the admin panel, go to Collections → households → New record.
  2. Set name to something friendly, e.g. "Our Collection".
  3. Set member_sub_1 to the PocketID sub value for the first household member. Find this in the PocketID admin panel under Users — it is the "ID" shown on each user's profile.
  4. Optionally set member_sub_2 for the second household member, and viewer_subs (as a JSON array, e.g. ["sub-value-here"]) for any view-only members.
  5. Save.

User roles come from PocketID groups, not from these fields — a user in the cup-owner group is an owner regardless of whether their sub appears here. The member_sub_* and viewer_subs fields are used by the app to find which household a user belongs to, so that owned-cups data is scoped correctly.
