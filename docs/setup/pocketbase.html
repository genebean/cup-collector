PocketBase Setup

This page covers three tasks: creating the initial superuser, verifying that migrations ran correctly, and creating your household record. Migrations handle all schema creation automatically — you should not need to create collections by hand.

Migrations run automatically
The migrations in pocketbase/migrations/ create all collections and set access rules on first start. Check the PocketBase logs to confirm they applied cleanly before proceeding.

Step 1 — Create the initial superuser

PocketBase requires at least one superuser before it will accept API requests. The recommended approach is via the CLI — it works while the server is running and doesn't require a browser.

SSH into your server and run:

sudo systemctl exec cup-collector-pb -- \
  pocketbase superuser create YOUR_EMAIL YOUR_PASSWORD \
  --dir=/var/lib/cup-collector-pb/pb_data

Alternatively, use the upsert subcommand (create-or-update) if you want the command to be idempotent — safe to run again if you're unsure whether the account already exists:

sudo systemctl exec cup-collector-pb -- \
  pocketbase superuser upsert YOUR_EMAIL YOUR_PASSWORD \
  --dir=/var/lib/cup-collector-pb/pb_data
Save these credentials
Store the email and password in your password manager. You'll also need them in the app's .env file as POCKETBASE_ADMIN_EMAIL and POCKETBASE_ADMIN_PASSWORD so the Next.js server can write records on behalf of users.

If you prefer the browser UI: open an SSH tunnel (ssh -L 8090:localhost:8090 yourserver), then visit http://localhost:8090/_/ and follow the on-screen prompt to create an admin account.

Step 3 — Create the household record

The household record links your PocketID user groups to owned-cup data in PocketBase. The key field is group_slug — a short identifier that the app uses to match PocketID groups to a household at sign-in.

The convention is:

  • Choose a slug, e.g. smith-family
  • Create PocketID groups named cup-collector-smith-family-owner and/or cup-collector-smith-family-viewer
  • Set group_slug = "smith-family" on the household record in PocketBase

The cup-collector- prefix namespaces these groups so they don't collide with groups from other apps sharing the same PocketID instance. The app ignores any group that doesn't start with this prefix.

Users in the -owner group can mark cups as owned, edit condition info, and run imports. Users in the -viewer group can browse the collection but cannot make changes. A user can belong to groups for more than one household — the app will show a switcher in Settings.

Option A — Script (recommended)

The dev shell provides a create-household script. Open an SSH tunnel so your local PocketBase URL points at the production instance, then run from inside nix develop:

# On your local machine — forward production PocketBase to localhost:8090
ssh -L 8090:localhost:8090 yourserver

# In another terminal, enter the dev shell and run the script
# Override the URL and credentials to target production instead of dev
POCKETBASE_URL=http://localhost:8090 \
POCKETBASE_ADMIN_EMAIL=admin@example.com \
POCKETBASE_ADMIN_PASSWORD=yourpassword \
  create-household --name "Our Collection" --slug our-collection

The script checks for an existing record with the same slug before creating, so it is safe to run more than once. It prints the resulting record ID and a reminder about the PocketID group names to create.

For local development, the credentials are loaded from app/.env.local automatically — no need to set them explicitly:

create-household --name "Test Household" --slug test-household

Option B — Browser admin UI

  1. Open an SSH tunnel: ssh -L 8090:localhost:8090 yourserver
  2. Open http://localhost:8090/_/ and log in.
  3. Go to Collections → households → New record.
  4. Set name to something friendly, e.g. Our Collection.
  5. Set group_slug to your chosen slug, e.g. our-collection.
  6. Save.

Step 4 — Create PocketID groups

In the PocketID admin UI, create one or both groups for your household slug. Using slug our-collection as an example:

  • Name (the value in the groups OIDC claim): cup-collector-our-collection-owner
  • Friendly Name (display only): e.g. Cup Collector — Our Collection — Owner

Repeat for the viewer group if needed: cup-collector-our-collection-viewer.

Add each user to the appropriate group. On their next sign-in (or after signing out and back in), the app will resolve their household automatically.

Security model

PocketBase is not exposed publicly — the NixOS module does not create an nginx vhost for it by default. All browser traffic reaches PocketBase through the Next.js /api/pb proxy, which requires a valid signed-in session. The "unrestricted read" rules set by migrations are only reachable by traffic that has already passed the session check in the proxy.

Writes are admin-only in PocketBase. All mutations go through Next.js server-side API routes that authenticate with the admin credentials — the browser never gets write access to PocketBase directly.

Accessing the admin UI after setup
Use an SSH tunnel whenever you need the admin UI:

ssh -L 8090:localhost:8090 yourserver

then open http://localhost:8090/_/ in your browser.
