Section 01

Project Overview & Goals

Two people collect location-specific Starbucks mugs and cups from the You Are Here, Been There, and related series. After years of collecting they have so many cups they can't reliably remember what they own. This app solves that problem — at home for browsing, and on the road when a new cup is in front of them.

Primary Use Cases

  • Open the app while traveling and immediately see which cups from the current area are still needed
  • Find nearby Starbucks stores that might carry those cups, and navigate to them via Apple Maps
  • Browse the full collection by series, country, or owned/needed status
  • Mark a cup as owned — both phones update instantly without any manual syncing
  • Quick-check any city: "do we already have the Tokyo cup?"
  • Let family members (view-only) browse the collection

Non-Goals

  • Public sharing or social features — private household app
  • E-commerce or purchase tracking
  • Native iOS App Store distribution
  • Android-specific optimization (works in any browser; not a priority)
ℹ️
Design Philosophy

Self-sovereign by default. Every core component runs on infrastructure the owner controls. The only external dependency for core functionality is Google Places API for nearby Starbucks discovery — acceptable because it is a read-only location lookup, not a data custody concern.

🤖
Note for Agents & Claude Code

The owner is not an app developer. Prioritise clarity, long-term maintainability, and self-sufficiency over cleverness. When in doubt: self-hosted over cloud, open-source over proprietary, simple over clever, explicit over magic. Comment generously — future maintenance may be done by an AI agent without full context.

All tooling (node, npm, npx, ts-node, pocketbase) must come from the Nix dev shell. Never assume or suggest installing tools on the host system directly. See Section 03 and Section 10 for detail.

Section 02

User Roles

Three roles exist. Role is determined entirely by the user's PocketID group membership — no in-app role field is required. The groups claim is included in the OIDC token (via the groups scope) and resolved server-side by roleFromGroups() in src/lib/roles.ts.

RoleWhoCapabilities
Owner owner Primary collector (Gene) Full access: browse, mark cups owned/unowned, upload own photos, run data imports. Must be a member of the cup-owner PocketID group.
Collaborator collaborator Spouse / co-collector Same as owner — full read and write. Must be a member of the cup-collaborator PocketID group. Also the person who curates the cup catalog CSV.
Viewer viewer Family / friends Read-only: browse catalog, see owned/needed status, use map and nearby stores. Cannot mark cups owned or make any changes. Must be a member of the cup-viewer PocketID group.
⚠️
Role Enforcement is Two-Layer — Both Are Required

The UI hides write controls from viewers. Every mutating API route also enforces the same restriction by calling resolveRole(session.user.groups) and returning 403 for viewer-role users. PocketBase write rules are admin-only — all mutations go through Next.js server-side routes that authenticate with the admin email/password. Browser clients never get direct write access to PocketBase.

Section 04

Authentication & User Model

Authentication is handled by PocketID (self-hosted OIDC provider). Auth.js (next-auth v5) acts as the OIDC client in the Next.js app. Each user has their own PocketID account. Role is determined entirely by the user's PocketID group membership — no database lookup required. A household record in PocketBase links users to a shared cup collection for data scoping (filtering owned_cups queries), but it plays no part in role resolution.

Auth.js OIDC Configuration

// app/auth.ts
import NextAuth from "next-auth"

export const { handlers, signIn, signOut, auth } = NextAuth({
  providers: [{
    id: "pocketid",
    name: "PocketID",
    type: "oidc",
    issuer: process.env.POCKETID_ISSUER_URL,
    clientId: process.env.POCKETID_CLIENT_ID,
    clientSecret: process.env.POCKETID_CLIENT_SECRET,
    // "groups" scope required — PocketID includes group membership in the OIDC token
    authorization: { params: { scope: "openid profile email groups" } },
  }],
  callbacks: {
    authorized({ auth: session, request }) {
      if (!session?.user) return false              // not signed in → to /sign-in
      const groups = session.user.groups ?? []
      const knownGroups = [ROLE_GROUP_OWNER, ROLE_GROUP_COLLABORATOR, ROLE_GROUP_VIEWER]
      if (!groups.some(g => knownGroups.includes(g)))
        return Response.redirect(new URL("/access-denied", request.url))
      return true
    },
    async jwt({ token, profile }) {
      if (profile) {
        token.pocketIdSub = profile.sub           // stored for household lookup
        token.groups = profile.groups ?? []       // stored for role resolution
      }
      return token
    },
    async session({ session, token }) {
      session.user.pocketIdSub = token.pocketIdSub
      session.user.groups = token.groups          // exposed for roleFromGroups()
      return session
    }
  }
})

Role Resolution Flow

On every authenticated request the app resolves the current user's role directly from the groups claim stored in the session token — no database lookup required:

  • Session token contains groups[] from the PocketID OIDC token
  • roleFromGroups(session.user.groups) returns owner | collaborator | viewer | none
  • Group names default to cup-owner, cup-collaborator, cup-viewer — configurable via env vars
  • No matching group → access denied, redirect to an "access not granted" page

The household record is looked up at sign-in by matching the user's PocketID group slug (e.g. smith-family-owner → slug smith-family) against the group_slug field on each household record. A user may belong to groups for multiple households; all matching households are resolved and stored in the JWT. The Settings screen shows a switcher when more than one household is resolved. This lookup is for data scoping, not for role determination.

PocketBase Access Rules

All three collections have unrestricted read rules ("") and admin-only write rules (null) at the PocketBase level. This is intentional — browser clients do not hold PocketBase auth tokens, so PocketBase's @request.auth rules cannot be used for enforcement. Security is provided at two other layers:

  • Network layer: PocketBase is not publicly exposed by default — all browser traffic goes through the authenticated Next.js /api/pb proxy, which requires a valid Auth.js session before forwarding.
  • API layer: All mutating routes authenticate with PocketBase admin credentials server-side and check the user's role before proceeding.
ℹ️
Ownership as Existence

A cup is "owned" if and only if a record exists in owned_cups with the matching household ID and cup ID. There is no status field. To un-own a cup, delete the record. This keeps queries simple and eliminates invalid state.

Section 07

Feature Specifications

Location Awareness

On first load the app requests navigator.geolocation permission. If granted:

  • Map centers on current location with a white "you are here" circle marker with a blue outline
  • Browse list sorts cups nearest to current position first (unowned before owned); no radius cutoff
  • "Near Me" filter chip auto-applied on the Browse screen
  • Nearby Starbucks lookup triggered automatically via the Google Places proxy route

If the user declines location permission the app falls back gracefully — map shows world view, Browse shows all cups unsorted. No broken states or error messages.

Realtime Sync

PocketBase provides realtime subscriptions via Server-Sent Events. The app subscribes to owned_cups on mount. When either member marks a cup owned or removes it, all connected devices update within 1–2 seconds without a page refresh.

// Subscribe to owned_cups changes for this household
pb.collection('owned_cups').subscribe('*', (e) => {
  if (e.record.household_id === currentHouseholdId) {
    // Invalidate the owned cups query — React Query refetches automatically
    queryClient.invalidateQueries(['owned_cups', currentHouseholdId])
  }
})

Owned / Needed Toggle

  • Not owned → green button "✓ Mark as Owned" → creates a record in owned_cups
  • Owned → red button "Remove from Collection" → deletes the record
  • Optimistic UI — button reflects new state immediately, rolls back if server returns an error
  • Button not rendered at all for viewer-role users

Flake Outputs

The flake exports three outputs, all defined in flake.nix:

  • packages.default — The Next.js app built with buildNpmPackage using nodejs_24. Produces a standalone output suitable for running as a Node process. npmDepsHash must be recomputed after any package-lock.json change (run nix build — it will fail and print the correct hash).
  • packages.migrations — The PocketBase migrations directory, exported as a Nix path. Passed to services.cupCollector.migrationsDir in the consuming host's config so migrations run on startup.
  • devShells.default — Development shell providing nodejs_24 and python3 (for docs-serve), plus shell hook aliases: pb-serve, pocketid-serve, dev-next, import-cups, gen-auth-secret, docs-serve, and check. PocketBase and PocketID run as OCI containers via podman; TypeScript tooling comes from npm devDependencies, not host packages.
  • nixosModules.default — System-independent NixOS module that wraps nixos/module.nix and pre-wires appPackage to this flake's own built output.

Section 11

Data Loading Strategy

The cup catalog is maintained by the collaborator in a spreadsheet (Google Sheets or Excel) and imported via CSV. For large catalog updates (new series, bulk corrections), the owner runs the CLI import script from inside the dev shell. For incremental updates (a handful of new cups), the owner or collaborator can use the web UI at /admin/import without needing a dev shell.

CSV Column Format

city,region,country,country_code,series,year,lat,lng,image_url,notes
San Francisco,California,United States,US,You Are Here,2021,37.7749,-122.4194,https://...,
Tokyo,,Japan,JP,You Are Here,2019,35.6762,139.6503,https://...,
London,England,United Kingdom,GB,Been There,2020,51.5074,-0.1278,https://...,

The image_url column accepts a public URL. The import script downloads the image and stores it in PocketBase file storage. Images are not linked externally after import — the URL is stored only in image_credit for attribution.

Import Script — scripts/import-cups.ts

  • Must be run inside the Nix dev shell: enter with nix develop first
  • Run: import-cups --file cups.csv (shell alias defined in flake.nix)
  • Dry-run preview: import-cups --file cups.csv --dry-run
  • Upsert logic: match on city + series + year — update if exists, create if not
  • Safe to re-run at any time — will not duplicate records
  • Images downloaded and saved to PocketBase; source URL stored in image_credit
  • Web UI alternative: /admin/import — same upsert logic, accessible to owner and collaborator roles

Section 13

Documentation Structure & Requirements

All documentation lives in docs/ as pure HTML files served via GitHub Pages. No static site generator. A GitHub Actions workflow deploys automatically on every push to main that touches docs/.

🤖
HTML-Only — No Generator, No Markdown

All files in docs/ are plain HTML. No Jekyll, Hugo, MkDocs, or any other tool. GitHub Pages serves them as static files with zero build step. Agents edit HTML directly. Markdown is not used for documentation. Navigation between pages uses standard <a href> links. The dev shell provides a docs-serve command that serves the docs locally at localhost:4000 for preview.

Docs are tailored for this specific owner but must be complete enough for a different person to run their own self-hosted copy. End-user docs come first in the navigation so the owner can link family members to them directly.

Section 15

Decision Log

All key decisions made during planning. Preserved for context. Do not override without updating this document.

Why a PWA instead of a native iOS app?
For a private two-person app the PWA hits every requirement at a fraction of the build effort. No App Store account, no review process, instant updates. If native feel ever becomes a priority, migrating to React Native is a natural next step — the PocketBase backend carries over entirely.
Why PocketBase instead of Firebase or Supabase?
Self-sovereignty. PocketBase is a single Go binary with no external dependencies. Data stays on hardware the operator controls. Firebase and Supabase are both third-party data custodians.
Why Auth.js instead of PocketBase's built-in auth?
PocketBase's built-in auth cannot act as an OIDC relying party — it cannot consume an external OIDC provider. Auth.js handles the OIDC client flow against PocketID. PocketBase is used for data only.
Why Google Places instead of OpenStreetMap/Overpass?
Overpass was the first preference (open, no key). Accepted trade: Google Places provides more reliable and complete Starbucks data, especially outside major cities. Key stays server-side only. Personal use will almost certainly remain in the free tier. Not a data custody concern — read-only location lookup.
Why separate accounts with a household record instead of one shared account?
Each person logs in with their own PocketID identity. This enables correct attribution, supports viewer accounts without sharing credentials, and allows each person to have their own session and sign-in experience.
Why Leaflet + OpenStreetMap for the map?
Fully open source, no API key for tile rendering, works well as a PWA. Google Maps JavaScript API and Mapbox both require keys and have usage costs. The Google Places API key is used only for the Nearby Search server-side call — not for rendering the map.
Why a NixOS module in the app repo rather than inline host config?
Keeps deployment config co-located with the code it deploys. The dots repo adds the app repo as a flake input and imports the module. Updating the app and its deployment config is a single commit in one repo; the dots repo bumps the flake input to pick up changes.
Why is PocketBase containerised but Next.js is not?
PocketBase is not straightforward to build from source in nixpkgs, so a pinned OCI image is the most reliable declarative approach. Next.js builds cleanly with buildNpmPackage and runs as a plain Node process — no container overhead, integrates more naturally with NixOS systemd service management.
Why pure HTML docs with no static site generator?
Markdown is too limiting for rich documentation layout. Agents can create and edit HTML directly without learning a generator's conventions or build pipeline. GitHub Pages serves static files natively with no build step. Maximally portable and maintainable by both humans and agents.
Why is the Nix dev shell the tooling contract?
The owner runs NixOS and manages infrastructure declaratively. Installing Node globally would be inconsistent with that philosophy and could cause version conflicts with other hosted services. The dev shell provides a reproducible, version-pinned environment for all app development and maintenance tasks without polluting the host.
Why CSV import instead of live Google Sheets sync?
For this round, simplicity wins. The collaborator maintains the sheet, exports CSV, the owner runs the import script. No Google Sheets OAuth, no scheduled jobs, no network-dependent failure modes. Can be upgraded to automated sync later — the import script is already idempotent.
Why drive permissions from PocketID groups and support multiple households per user?
Diverged from original spec. Two goals drove this change together: (1) drive as much access control as practical through PocketID rather than the app's own database, and (2) allow a user to belong to more than one household (e.g. helping curate a friend's collection) without schema changes.

The original design stored PocketID sub values in member_sub_1, member_sub_2, and viewer_subs on the household record. Role and household membership were both derived by looking up those fields at sign-in. The replacement: household records carry a single group_slug field (e.g. smith-family). PocketID groups follow the naming convention cup-collector-{slug}-{role} (e.g. cup-collector-smith-family-owner). The cup-collector- prefix namespaces groups so other apps sharing the same PocketID instance don't interfere — the app silently ignores any group without this prefix. At sign-in the JWT callback reads the groups claim, strips the prefix and role suffix to recover the slug, and queries PocketBase for matching households. Role comes from the suffix; household comes from the slug match. All matched households are stored in the JWT as householdMemberships. Adding a user to a second household is just a PocketID group assignment — no database changes needed.

When more than one household is resolved, the Settings screen shows a household switcher. Switching is instant: Auth.js session.update() swaps the active householdId without a sign-out. The net result: PocketID is the single source of truth for who has access to what, and the app's database holds only cup data.
Why Places API (New) Text Search instead of the legacy Nearby Search?
The legacy nearbysearch endpoint is deprecated and was returning inconsistent results. The Places API (New) searchText POST endpoint is the current recommended approach. The API key is passed in a request header (X-Goog-Api-Key) rather than a URL query parameter, which is more secure and easier to audit. Google's locationBias is a soft hint rather than a hard radius, so a haversine post-filter enforces the actual radius constraint.
Why darkMode: class via @variant dark (&:is(.dark *)) in Tailwind v4?
The app has deliberate brand colors (header green, gold accents, bottom nav) that should never change between light and dark — they are fixed brand colors, not neutral surfaces. Class-based dark mode lets those elements stay constant while allowing neutral surfaces (cards, backgrounds, text) to adapt. The .dark class is toggled on <html> by the useUiTheme hook, which reads the ui_theme localStorage key (System / Light / Dark). System mode follows prefers-color-scheme. The UiThemeInitializer component (rendered inside Providers) applies the class on every page load without requiring the Settings page to be open.
Why is the unit test coverage scope limited to src/lib/?
React pages, components, hooks, and Next.js API routes require either a browser environment (jsdom) or extensive mocking of framework internals (useRouter, useSession, React Query, PocketBase) to unit test. That mocking surface creates false confidence — mocked tests can pass while the real integration fails. The e2e suite (Playwright + real PocketBase) covers all UI behaviour meaningfully. Vitest is scoped to src/lib/, which contains only pure functions with no framework dependencies. This keeps coverage numbers honest: a high percentage in src/lib/ actually means something. The rule: if a function needs React or a browser to run, it belongs in src/components/, src/hooks/, or src/app/ and is covered by e2e tests. If it is pure logic, it belongs in src/lib/ and gets a unit test.
